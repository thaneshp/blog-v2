If you’re familiar with GCP, you’ll know that the [roles/resourcemanager.projectIamAdmin](https://docs.cloud.google.com/resource-manager/docs/access-control-proj#resourcemanager.projectIamAdmin) role is used to grant project-level roles.

As a developer or engineer working on a project, you’d want this level of permission so that you can grant required IAM roles to service accounts and other members.

However, this predefined role is highly permissive, and it should be granted to limited members only.

---

## What is the risk?

Say you manage a GCP project, where there are different people responsible for different parts of the stack, i.e. Compute, Database, etc.

Each group may need a different set of permissions for provisioning their respective resources and typically additional permissions for granting roles.

For example, the Compute team might need the following roles for provisioning Google Kubernetes Engine (GKE) resources:

- `compute.admin` - for provisioning the cluster 
- `projectIamAdmin` - for assigning permissions to developers to access the cluster

The risk is: if you’re providing `projectIamAdmin` permissions to the Compute team, this also gives them permission to modify the project IAM policy, including granting powerful roles to themselves or others.

Fundamentally, this breaks the **principle of least privilege** and increases the **risk of privilege escalation**.

So the question is: **how can you give members the required level of permission to grant new roles while limiting which roles they are allowed to grant?**

---

## Solution

The solution is to delegate IAM administration with IAM Conditions.

The pattern is to grant `roles/resourcemanager.projectIamAdmin` conditionally, so the principal can only grant or revoke a specific list of approved roles.

This preserves the self-service workflow while reducing the blast radius of the delegated administrator identity.

<Callout type="note">
The core idea here is the concept of [least privilege](https://en.wikipedia.org/wiki/Principle_of_least_privilege).

The principal should have enough access to perform the required workflow, but not enough access to grant unrelated or more privileged roles.
</Callout>

---

## Implementation

The Terraform implementation looks something like below.

```HCL
resource "google_project_iam_binding" "grant_gke_roles" {
  project = "your-project-id"
  role    = "roles/resourcemanager.projectIamAdmin"

  members = [
    "serviceAccount:your-service-account@your-project-id.iam.gserviceaccount.com",
  ]

  # Here is where you scope the permissions to only grant the required roles.
  condition {
    title       = "gke_roles_only"
    description = "Allow this principal to grant or revoke only approved GKE roles."
    expression  = <<EOT
api.getAttribute(
  'iam.googleapis.com/modifiedGrantsByRole',
  []
).hasOnly([
  'roles/container.developer',
  'roles/container.clusterViewer'
])
EOT
  }
}
```

Once applied, the service account will be able to grant the explicitly listed roles, but does not have permissions for granting anything else.

The example above uses GKE roles, but the same pattern can be applied to other delegated workflows where a CI/CD identity needs to grant a narrow set of approved roles.

---

<Callout type="info" title="Real World Use Case">
A similar pattern came up in one of the projects I worked on involving Cloud SQL.

In this project, we had to create a CloudSQL database and give developers self-service creation of databases via GitHub PR process.

To ensure developers (and/or other requesters) couldn’t elevate their permissions or apply unrelated roles, we scoped the CI service account used by the Github Action workflow to only grant permissions for connecting with the instance, i.e. `roles/cloudsql.client`.

Whilst it should be the responsibility of the approver to validate that the requester hasn't added anything outside the scope of the self-service workflow, things can slip through the cracks and this adds an added level of precaution.
</Callout>
