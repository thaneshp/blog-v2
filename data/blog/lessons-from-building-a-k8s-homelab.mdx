## Introduction

I’ve been working with Google Kubernetes Engine (GKE) for some time now, and whilst doing so, I gained a solid grasp of how Kubernetes operates at the platform level.

With the cloud offerings that are available today, it’s become easier than ever to build Kubernetes clusters (and other bits of infrastructure) without having to go too deep into how the underlying infrastructure operates.

As such, I built a Kubernetes homelab so that I could learn and experiment with some of its inner-workings from the ground-up.

This post goes over my initial design process and the lessons I learnt post-build.

---

## Initial Approach

There were a few overarching principles that guided my initial decision-making process:

1. Cost-efficient: it should be cost-efficient to build/run.
2. Scalable: I should be able to add nodes easily.
3. Maintainable: I should be able to make changes easily.

The diagram below shows how I initially thought of the architecture. In this section, I’ll share some of my thought process around each of the components I considered.

### Hardware

Everything stems from the hardware, so this was the first piece of the puzzle I had to think about.

I already had a Raspberry Pi 4 at home, so I just needed another machine that has enough resources to run Proxmox.

Now, I could have gone with another Raspberry Pi, but that wouldn’t have let me install a hypervisor and run VMs comfortably. So, I opted for a micro SFF PC which I found used online for ~$150 AUD.

### OS

For the most part, Kubernetes works best on Linux. 

There were two options I was tossing up between:

- Talos Linux - a minimal operating system for Kubernetes
- Ubuntu Server - a general-purpose, open-source Linux OS.

I went with Ubuntu Server for 2 key reasons: 
- There was known support for Raspberry Pis
- It allowed for SSH, which I wanted for Ansible and debugging if necessary.

### Infrastructure Automation

I knew from day 1 I wanted to have everything running using Infrastructure-as-Code in some shape or form for a few reasons:
- Visibility - I know how my infrastructure looks without having to guess.
- Consistency - VMs and configurations are the same across the stack.
- Maintainability - I can easily change any part of the stack.

Terraform and Ansible were the obvious choices for me here.

- Terraform - because of my familiarity with the tool and provider availability.
- Ansible - because of its known prevalence within the industry.

If you're interested, you can check out the code [here](https://github.com/thaneshp/homelab).

### Kubernetes Flavour

There were several Kubernetes flavours available for building homelabs, e.g. [minikube](https://minikube.sigs.k8s.io/docs/) & [K3s](https://k3s.io/).

I wanted a flavour which was used for **real** workloads/environments, so I chose [kubeadm](https://kubernetes.io/docs/reference/setup-tools/kubeadm/).

## Learnings

Once I had gathered an idea of all the components, I started putting the pieces together; and this is where the real learning took place.

Throughout the process, I learnt lots of little things along the way, but ultimately they fit into 3 overarching lessons I describe below.

### Lesson 1: Invest in maintainability early on

When I first started out on this journey, all I wanted to do was get to the state where I could start installing `kubeadm` and get the cluster up and running.

However, I took a step back, thought about the problem, and laid the groundwork for what I was about to build.

Irrespective of this, every time an issue popped up, it would be tempting to take shortcuts to move ahead; but ultimately, I knew this would cause re-work for me down-the-line.

Fundamentally, I knew this was going to be a long-term project and something I wanted to build upon over years and years. This meant ensuring the system was maintainable from day 1, i.e. by:

- Creating re-usable VM templates
- Utilising Infrastructure-as-Code
- Documenting everything

This creates more upfront effort, but significantly simplifies the process later on.

### Lesson 2: Understand the limitations of the tools you’re working with 

Every technology has its limitations.

I went into building this cluster head first, after reading a few blog and Reddit posts.

I knew all the tools I needed, but I didn’t understand each of their limitations; which I ended up paying the price for.

<Callout type="info">
Some issues I encountered:
1. **The differences between how each of the Terraform providers connect with Proxmox**
  - I initially went with the [bpg](https://registry.terraform.io/providers/bpg/proxmox/latest/docs) provider, which used SSH and led to connectivity issues. I switched to [Telmate](https://registry.terraform.io/providers/Telmate/proxmox/latest/docs) (API) and didn’t encounter any issues.
2. **Creating VMs without having the OS run installation steps each time**
  - When you create VMs with just the ISO, it runs the installation steps each time. This is solved by creating a template to reference when creating a new VM.
3. **Difference between allocated memory and usable memory on VMs**
  - Allocating 2GB of memory when creating a VM doesn’t guarantee that the OS might consume some memory. You might need to allocate more than that if your requirements are 2GB.
</Callout>

It’s not possible to account for every scenario; and some things just have to be learnt and experienced as you piece things together. For example, I found countless examples where people ran K8s clusters on Raspberry Pis, but as I tried to run `kubeadm`, it failed as the memory `cgroup` was disabled by default.

In any case, the lesson is that the initial effort to lay out potential limitations will likely save you time and frustration down the road.

### Lesson 3: Design around the end state

When we’re building technology, it’s easy to get carried away in the tools.

Sometimes we become fixed on the how to build something, instead of the why and what. When this happens, we might spend more time than necessary trying to debug or troubleshoot something when you could just cut your losses and move on with another approach.

<Callout type="info">
In my experience, this was:
- Troubleshooting SSH errors with the `bpg` Terraform provider, instead of moving on to another available provider.
- Investigating how to enable memory `cgroups` on Raspberry Pis, instead of just creating another VM on Proxmox.
</Callout>

So ultimately, it’s important to remember why you’re building something so that you’re not tied down to any specific tool/technology and can make adjustments as necessary.

However, knowing when to *pivot* or *persevere* depends entirely on the scenario at hand.

## End Result

This is what the end/current state looks like.

It’s more or less similar to my initial approach except I moved the control node into the hypervisor, i.e. because of the complications I experienced with the Raspberry Pi.

But for a homelab, it’ll do just fine…
