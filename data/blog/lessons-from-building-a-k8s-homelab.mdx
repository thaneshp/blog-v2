## Introduction

I’ve been working with Google Kubernetes Engine (GKE) for some time now, but have always felt that I didn’t have a solid grasp on what’s happening under-the-hood.

With the managed cloud offerings that are available today, it’s become easier than ever to build Kubernetes clusters (and other bits of infrastructure) without having to go into the weeds of things.

So, for my learning, I decided to build a Kubernetes cluster at home so that I could learn some of its inner-working and later use it to run experiments locally.

As I started to put the pieces together, I gained a much greater appreciation for all the abstractions that cloud hyperscalers provide; and I’ll share why in this post.

---

## Initial Approach

I had an initial thought and decision-making process behind how I’d build out this cluster. There were a few overarching principles that guided my decision-making process:

1. Cost-efficient: it should be cost-efficient to build/run.
2. Scalable: I should be able to add nodes easily.
3. Maintainable: I should be able to make changes easily.

The diagram below shows how I initially thought of setting things up. In this section, I will share some of my thought process around each of the components I thought about.

### Hardware

Everything stems from the hardware, so this was the first piece of the puzzle I had to think about.

I already had a Raspberry Pi 4 at home, so I just needed another machine that has enough resources to run Proxmox.

Now I could have gone with another Raspberry Pi, but that wouldn’t have had enough juice to let me install a hypervisor and run VMs comfortably. So, I opted for a micro SFF PC which you can find used online for ~$150 AUD.

### OS

For the most part, Kubernetes is suited to run on Linux. 

There were two options I was tossing up between:

- Talos Linux - a minimal operating system for Kubernetes
- Ubuntu Server - general purpose open-source Linux OS.

I decided to go with Ubuntu Server for 2 key reasons: 

1. There was known support for Raspberry Pi
2. It allowed for SSH which I wanted for Ansible setup and debugging if necessary.

### Infrastructure Automation

I knew from day 1 I wanted to have everything running using Infrastructure-as-Code in some shape or form for a few reasons:

- Visibility - I know how my infrastructure looks without having to guess.
- Consistency - VMs and configuration are the same across stack.
- Maintainability - I can make easily make changes to any part of the stack.

Terraform and Ansible were the obvious choices for me here.
- Terraform - because of my familiarity with the tool plus available providers for Proxmox.
- Ansible - because of its known usage within the community for configuring VMs.

### Kubernetes Flavour

There were several Kubernetes flavours available for building homelabs, e.g. minikube & K3s.

I wanted a flavour which was used for **real** workloads/settings, so I chose `kubeadm`. 

As for the version, `v1.35.0` is the latest Kubernetes version as of writing, so it was the obvious choice.

## Learnings

Once I had gathered all the details, I had a good idea of the plan in my head, but as I started to put things together, challenges popped-up which I didn’t expect.

However, each of those challenges provided some overarching lessons that I could take away.

### Lesson 1: It's worth setting things up properly from the get-go

When I first started out of this journey, all I wanted to do was get to state where I could start installing `kubeadm` and get the cluster up and running.

I knew I had to take a step back and set things up properly.

Fundamentally, I knew this was going to be a long-term setup and something I want to build upon over years and years. This means ensuring the system is maintainable.

To ensure it is maintainable, means setitng things up right from the get-go, e.g.

- Creating VM templates
- Utilising Infrastructure-as-Code
- Documenting everything

This creates more upfront effort, but simplifies the process later on.

### Lesson 2: Understand the limitations of the tools you’re working with 

Every technology has it's limitations.

I went into building this cluster head first, after reading a few blogs and reddit posts.

I knew all the tools I needed, but I didn't spend time understanding each of their limitations; which I ended up paying the price for.

Some example of this are:
1. Researching the different available Proxmox Terraform provider
2. How to create VMs without having the OS installed each time
3. Difference between allocated memory and available memory

It's not possible to account for every scenario, some things just have to be learnt and experienced as you piece things together.

However, the initial effort of laying out potential limitations would save time and frustration down the road.

### Lesson 3: Always keep the end state in mind

When you're putting together a cluster or building any technology, it's easy to get carried away in the tools.

If you have a fixed idea of **how** you are going to build something, then you might spend more time than necessary trying to debug something when you could just cut your losses and move on to another tool.

So, it's important to remember why you're building something so that you're not tied down to any specific tool/technology.
